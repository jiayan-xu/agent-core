## 一、三方观点摘要

### 1. Memoria 部署与运维专家
- G3/G4 的 memoria MCP schema 缺陷已在源码修复（`1fdd4a5`：给 `evolution_rollback`/`evolution_log_query` 补 `namespace`，对齐 `memory_evolve` 模式，`cargo check` 通过），但运行实例是 v0.3.0 旧二进制。
- **仅为了闭合 G3/G4 单独 redeploy 一次活服务 ROI 低** —— 因为 BoN 不过验收线，门反正开不了。
- 建议：**并入「BoN 复测过线后的常规发版」一并做 redeploy**，而非现在单独跑。
- 附安全步骤：停 `memoria-server.exe`（防 os error 5，改 Rust 源前必停）→ 禁桌面 Memoria.lnk 看门狗 → `cargo build --release` → 验证 G3（rollback 落 `rolled_back` 负样本）+ G4（`agent_registry` 清点，当前=15 查孤儿）→ 旧 exe 备份秒级回退。**难度 P1，约 1 人日**。

### 2. BoN 推理时计算专家（挖出关键根因）
- **根因发现**：生产选择器 `score_by_judge`（`llm.rs:304-314`）用「对 N 候选各给绝对分」弱提示；而 eval 测量 judge（`eval_bon.rs:89`）用 rubric 细粒度提示 —— **两者口径不一致**，正是「8 被当最优压过 10」的元凶（selection bug 真因，非生成质量差）。
- 推荐顺序：**E(解耦) → A(修选择器·根因) → D(放宽门) → B/C(调参)**。
- 强调：调参（B/C）必须在修 A 之后才有意义，否则方差大也选不准。

### 3. agent-core 架构与门纪律专家
- BoN 已证伪，应降为**观测指标**，硬门改「selector 修复后重测 Δpp≥0 且不回归」。
- G3/G4 源码修 ≠ 复验通过；须 redeploy 运行时验证才算过（边界：源码修好+redeploy 验证通过才算）。
- 开闸顺序：**技能库 → LATS → MultiAgent Compose**（技能库最自包含先开；LATS 需预算熔断 live；MultiAgent 依赖前两者 + 圆桌）。

- **阶段0（立即可决策，0d）**：落 **E + D** —— BoN 从开闸硬门**降级为观测指标**；三大项 gate 改为「G1–G4 全绿 + 各 item DoD 全绿」；BoN 自身 gate 另立为「selector 修复后重测 Δpp≥0 且不回归」。→ 三大项不再被 BoN 卡死。
